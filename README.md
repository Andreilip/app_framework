Каркас приложения с гексагональной архитектурой на 
Python.
Приложение представляет собой набор таблиц (пока одна), к которым 
можно обращаться через эндпоинты и выполнять CRUD операции,
либо какие-то операции связанные с бизнес логикой.

Смысл данной архитектуры - отделить бизнес логику приложения
от инфраструктуры. 
Есть следующие слои:
- Бизнес-логики. В нем только объекты и операции с объектами,
заданными бизнес правилами. (папка domain)
- Приложения. В нем описывается логика взаимодействия слоев
на уровне интерфейсов. То есть что нужно сделать, без реализации
как делать. Описываются входные и выходные структуры данных
как объекты языка Python. (папка application)
- Адаптеры. В данной папке описываются все реализации 
пользовательских сценариев (как делать) и их передача 
в пользовательские или программные интерфейсы. (папка adapters)
- Инфраструктуры. В нем описаны инфраструктурные элементы, 
используемые приложением. СУБД, системы авторизации,
системы кэширования, очереди и т.д. Безотносительно
пользовательских сценариев и логики 
приложения (папка infrastructure)
- Пользовательские и программные интерфейсы. Описываются 
технические моменты реализации возможных интерфейсов
доступа к системе, например, API, CLI и т.д. Безотносительно
пользовательских сценариев и логики приложения(папка ui)


Приложение проектируется так:
- Пользователь (бизнес аналитик) придумывает какой-либо сценарий, 
например, добавление, удаление или увольнение сотрудника.
- Проектируется интерфейс пользовательского сценария в файле(папке)
\src\application\use_case_interfaces.py.
  - Формируются входные классы данных "DoSmthWithObject"InputDTO.
  - Формируются выходные классы данных "DoSmthWithObject"ResultDTO.
  - Формируются интерфейсы самих сценариев, реализация отсутствует
  "DoSmthWithObject"UseCaseInterface.
  - Здесь нам интересно, только что мы подаем на вход, на выход 
  и как вызвать данный метод. У интерфейсов всегда есть только 
  один метод execute.
- Проектируются объекты, которые участвуют в этом пользовательском
сценарии:
  - В файле (папке) src/domain/models.py прописываются их 
  свойства и методы в терминах бизнес логики. "Object". На этом
  этапе нужно перечислить как можно больше свойств и методов, чтобы
  избежать дальнейших переделок.
  - В файле (папке) src/domain/services.py прописываются функции, 
которые совершаются над объектами "DoSmthWithObject". Могут быть как
  без классов так и классы.
- Проектируется интерфейс работы с  данными в репозитории 
в файле (папке) src/application/repo_interfaces.py.
  - Мы не знаем какая именно будет инфраструктура и реализация,
  поэтому интерфейсы создаются максимально абстрактно,
  безотносительно технологии хранения данных.
  - Методы могут использовать классы из доменной области (агрегатов)
  - "Object"RepoInterface - наименование интерфейсов
- Проектируются реализации пользовательских сценариев на основе
их интерфейсов и интерфейсов репозиториев, а также методов доменной
области src/application/use_cases.py. Здесь описываются, какие 
именно методы с какими входными значениями и в каком порядке
мы будем вызывать, чтобы реализовать бизнес логику приложения.
  - Классы называются "DoSmthWithObject"UseCase
- Далее идет реализация репозиториев конкретной технологии для 
конкретного репозитория в src/adapters/repos.py. Адаптер позволяет 
проектировать логику приложения независимо от технологии. Следует
понимать, что адаптер уже использует конкретные методы технологии,
которые содержаться в папке src/infrastructure. 
  - Классы называются "TechnologyDatasourceObject"Repo
- Далее идет реализация контроллеров в адаптерах. В данном случае
это эндпоинты. При вызове эндпоинта создаются репозитории, необходимые
наборы входных данных и запускаются пользовательские сценарии. Также
автоматически через pydantic реализуется проверка входных типов данных
и сериализация выходных типов данных.
  - Создаются классы входных данных в pydantic (если нужно)
  "DoSmthWithObject"InputSchema
  - Создаются классы выходных данных в pydantic
  "DoSmthWithObject"ResultSchema
  - Создаются роутеры (эндпоинты) do_smth_with_object

Запуск приложения осуществляется: 
  - python main.py

Сбор всех зависимостей осуществлен в файл requirements.txt:
  - pip freeze >requirements.txt

Поставить зависимости на другой машине можно так:
  -	pip install -r requirements.txt
